# 答辩演示指南

> 本文档提供完整的答辩演示路径与配套话术，建议演示时长 8–10 分钟。

## 📝 详细演示步骤

### ① 登录与权限（针对老师"缺登录界面"意见）

**操作**：打开系统，登录页右侧点击 `admin` 一键填充

**话术**：
> 系统采用前后端分离的单页应用架构，登录采用密码哈希校验。系统支持 5 种角色：系统管理员、工程管理员、采购员、运维员、报废鉴定员。
> 我特意保留了一个冻结账户 `frozen`，演示禁用状态的提示。

**操作**：先尝试用 `frozen` 登录 → 提示"账户已被禁用"，再换 `admin`

### ② 首页仪表盘（针对老师"统计率要真实算出来"意见）

**操作**：登录后进入首页

**话术**：
> 这里展示的 8 项关键率指标，全部由数据库聚合计算得出，没有任何硬编码：
> - **设备在用率**：使用中设备数 / 总设备数
> - **设备完好率**：非故障非报废设备占比
> - **保养完成率**：周期内已保养设备 / 应保养设备
> - **工单关闭率**：已关闭工单 / 总工单
> - **MTBF**：平均无故障时间，由维修工单时间差计算
> - **MTTR**：平均修复时间，由停机时长聚合
> - **年故障率** 与 **累计运维成本**

**指给老师看**：30 天费用趋势折线图，是真实的保养记录与维修工单聚合。

### ③ 全生命周期视图（针对老师"没体现全生命周期"意见）

**操作**：左侧菜单点击"全生命周期"

**话术**：
> 这是这次改进新增的核心模块。每台设备的完整生命周期事件——从采购、投运、历次保养、维修、到最终报废——都会写入 `lifecycle_event` 流水表。
> 左侧是设备列表，每条都有寿命使用率进度条（设计寿命对比已用年限）。
> 右侧时间轴按时间倒序展示完整事件链，每条事件包含：操作类型、时间、操作人、关联费用。
> 顶部累计 TCO 实时聚合该设备的购置 + 保养 + 维修费用。

**操作**：选一台超过 80% 寿命的设备，展示它的完整事件链。

### ④ 报废技术鉴定（论文核心创新点）

**操作**：用 `scrap` 重新登录，或用 `admin` 直接进入"报废技术鉴定"

**话术**：
> 报废决策是论文最核心的章节。传统做法依靠经验拍脑袋，本系统基于 **TCO 全寿命周期成本模型** 进行量化决策。
> 待鉴定列表自动筛选出超过 60% 设计寿命的设备，按使用率倒序排列。

**操作**：点一台进入鉴定详情页

**话术**：
> 三栏布局：
> - **左侧**：设备完整档案——投运日期、已用年限、使用率、购置价、残值估算、年保养费、年维修费
> - **中间**：TCO 决策面板，对比"维修延寿"与"更换报废"两个方案
>   - 方案 A：修复一次性投入 + N 年保养费 + N 年维修费 - 残值
>   - 方案 B：新设备购置 + N 年保养费 + N 年维修费 - 旧设备残值
>   - 系统自动给出推荐方案 + 差额分析
> - **右侧**：鉴定结论提交，进入待审批工作流

**操作**：调整"预计修复费用"和"服役目标年限"，观察决策结果实时变化

**话术**：
> 这里有个**阈值前置触发机制**：当修复成本 ≥ 残值 × 阈值，或寿命使用率 ≥ 警戒线，系统会强制建议报废，避免"修不起的设备硬撑着"导致安全风险。

### ⑤ 阈值参数配置（论文 6.5 节）

**操作**：左侧菜单"阈值参数配置"

**话术**：
> 系统支持管理员动态配置 3 个核心阈值：
> - **修复成本/残值阈值**（30%–80%）：触发报废建议的临界点
> - **寿命警戒阈值**（60%–95%）：进入警戒状态的使用率
> - **市场折扣系数**（50%–100%）：残值估算的市场流动性折扣
> 所有修改都会写入 `sys_config` 表并记录审计日志。

**操作**：把"修复成本/残值阈值"从默认 60% 拉到 40%，回到刚才的鉴定页面，决策结果实时变了。

### ⑥ 统计分析

**操作**：左侧菜单"统计分析"

**话术**：
> 4 个分析维度：
> - **设备类别分布**：11 类设备的数量分布
> - **维护费用趋势**：12 个月的保养费 vs 维修费双 Y 轴折线
> - **故障率排行**：按分类的故障次数排行
> - **分类 TCO 对比**：购置 + 保养 + 维修的累计成本堆叠柱

**操作**：切换"近 30 天 / 近 3 月 / 近半年 / 近 1 年"，所有图表动态重算

### ⑦ 角色权限对比

**操作**：右上角点击退出 → 用 `purchaser` 重新登录

**话术**：
> 切换为采购员账号后，左侧菜单只剩"首页仪表盘"和"采购入库"两项，证明角色权限隔离生效。
> 所有操作均会记录到审计日志，可在管理员"审计日志"页面追溯。

## 🎯 老师可能问的问题与应答

**Q1：你的数据库为什么用 IndexedDB，论文里写的是 MySQL？**

A：论文采用 Spring Boot + MySQL 8 的真实后端架构，本演示版本为了在没有服务器的笔记本上能直接运行展示，使用浏览器 IndexedDB 模拟 MySQL。我严格按照论文 4.3 节的物理表设计实现了 17 张 ObjectStore，字段、类型、关系完全一致。生产环境只需要把 dbApi 替换为 RESTful 调用即可。

**Q2：TCO 公式的具体推导？**

A：参考论文 6.5.2 节，TCO = 一次性投入 + Σ(年保养费 × N) + Σ(年维修费 × N) - 残值回收。
- 维修方案：一次性投入 = 预计修复费
- 更换方案：一次性投入 = 新设备购置价
- 残值估算 = (购置价 - 已折旧值) × 市场折扣系数

**Q3：α 和 β 系数怎么定的？**

A：
- α 是气候系数，参考论文 5.3.1 节，分 5 个气候带：寒冷 1.30 / 严寒 1.20 / 温和 1.00 / 湿热 1.15 / 沿海 1.25
- β 是 AADT（年平均日交通量）系数，分 5 档：< 1万 取 0.85，1–3万 取 1.00，3–6万 取 1.15，6–10万 取 1.30，>10万 取 1.50
- 设备实际保养周期 = 标准周期 / (α × β)

**Q4：种子数据是不是太"完美"了？**

A：种子数据使用伪随机数生成，覆盖了正常、超期、临界、故障等多种状态。例如赣江一桥的悬索是温和气候 + 高 AADT，鄱阳湖大桥是湿热气候 + 中 AADT，松花江北桥是严寒气候——每座桥每个分类的设备处于不同生命周期阶段。

**Q5：审计日志能恢复数据吗？**

A：审计日志记录所有写操作（CREATE / UPDATE / DELETE），主要用于责任追溯，不做数据回滚。如果需要回滚能力，需要扩展为事件溯源（Event Sourcing）架构，这是后续优化方向。

## 💡 临场注意事项

1. **首次打开稍慢**：会初始化数据库 + 写种子数据，约 2-3 秒
2. **遇到 CDN 加载慢**：刷新一次即可，资源会缓存
3. **演示前清缓存**：浏览器 F12 → Application → Storage → Clear site data，可恢复初始数据
4. **多浏览器试一遍**：建议 Chrome / Edge，Safari 的 IndexedDB 实现略有差异
5. **答辩备份**：随身带 U 盘 + 提前在答辩教室电脑上预跑一次

## 📌 关键术语对照

| 中文 | 英文 | 含义 |
|------|------|------|
| 全寿命周期成本 | TCO (Total Cost of Ownership) | 设备从购置到报废的全部成本 |
| 平均无故障时间 | MTBF (Mean Time Between Failures) | 故障间隔时长 |
| 平均修复时间 | MTTR (Mean Time To Repair) | 单次维修平均耗时 |
| 年平均日交通量 | AADT (Annual Average Daily Traffic) | 桥梁运行强度核心指标 |
| 角色权限控制 | RBAC (Role-Based Access Control) | 基于角色的访问控制 |
