## 📝 详细演示步骤

### ① 系统架构展示

**话术**：
> 本系统采用前后端分离架构：
> - 前端：Vue 3 + Element Plus + ECharts
> - 后端：Spring Boot 3 + MyBatis-Plus + MySQL 8
> - 认证：JWT Token 无状态认证
> - 数据库：17 张表严格对应论文 E-R 设计

**展示**：打开项目目录，展示 frontend 和 backend 结构

### ② 登录认证（针对老师"缺登录界面"意见）

**操作**：浏览器访问 `http://localhost:5173`，点击"一键填充 admin"

**话术**：
> 系统采用 JWT Token 认证机制，登录时校验密码哈希（BCrypt）。
> 支持 5 种角色：系统管理员、工程管理员、采购员、运维员、报废鉴定员。
> 账户状态实时检查，冻结账户无法登录。

### ③ 首页仪表盘（针对老师"统计率要真实算出来"意见）

**操作**：登录后展示首页仪表盘

**话术**：
> 这里展示的 8 项关键率指标，全部由后端服务聚合计算：
> - **设备在用率**：使用中设备数 / 总设备数
> - **设备完好率**：非故障非报废设备占比
> - **年故障率**：本年度故障工单数占比
> - **保养完成率**：周期内已完成保养比例
> - **工单关闭率**：已关闭工单 / 总工单
> - **MTBF**：平均无故障时间（小时）
> - **MTTR**：平均修复时间（小时）
> - **累计运维成本**：保养 + 维修费用总和

**指给老师看**：30 天费用趋势图和待办事项列表

### ④ 全生命周期视图（针对老师"没体现全生命周期"意见）

**操作**：左侧菜单点击"全生命周期"

**话术**：
> 这是本次改进的核心模块。每台设备的完整生命周期事件都会写入 `lifecycle_event` 流水表：
> - 采购入库 → 投入运行 → 历次保养 → 历次维修 → 报废处理
> - 左侧设备列表显示寿命使用率进度条
> - 右侧时间轴按时间倒序展示完整事件链
> - 每条事件包含：类型、时间、操作人、关联费用

**操作**：选一台超过 80% 寿命的设备，展示它的完整事件链和累计 TCO

### ⑤ 报废技术鉴定（论文核心创新点）

**操作**：进入"报废鉴定"菜单

**话术**：
> 报废决策是论文最核心的章节。传统做法依靠经验判断，本系统基于 **TCO 全寿命周期成本模型** 进行量化决策。
> 待鉴定列表自动筛选出达到警戒条件的设备：
> - 寿命使用率 ≥ 警戒阈值（80%）
> - 故障次数 ≥ 警戒阈值（3 次）
> - 残值低于警戒线（20%）

**操作**：点击一台设备进入 TCO 决策详情页

**话术**：
> 三栏布局决策面板：
> - **左侧**：设备档案——投运日期、已用年限、购置价、残值估算、年保养费、年维修费
> - **中间**：TCO 对比计算
>   - 维修方案：修复费 + N 年保养费 + N 年维修费
>   - 更换方案：新设备价 + N 年保养费 - 残值回收
>   - 系统自动给出推荐方案 + 差额分析
> - **决策因素标记**：是否触达残值警戒线、修复警戒线

**操作**：调整"预计修复费用"和"服役目标年限"，观察决策结果变化

### ⑥ 阈值参数配置

**操作**：进入"系统配置"菜单

**话术**：
> 系统支持管理员动态配置 4 个核心阈值：
> - **残值警戒线**（0-50%）：设备残值低于此阈值进入待鉴定
> - **修复警戒线**（0-100%）：修复费占购置价超过此阈值进入待鉴定
> - **寿命警戒线**（50-100%）：使用年限超过设计寿命此比例进入待鉴定
> - **故障次数警戒线**（1-10）：累计故障超过此阈值进入待鉴定
> 所有修改实时生效，影响报废鉴定筛选条件。

**操作**：修改寿命警戒线从 80% 到 60%，回到报废鉴定列表查看新增设备

### ⑦ 统计分析

**操作**：进入"统计分析"菜单

**话术**：
> 4 个分析维度：
> - **设备分类分布**：11 类设备数量饼图
> - **TCO 对比分析**：各分类的平均购置、保养、维修、TCO 表格
> - 首页仪表盘还有费用趋势折线图和故障排行

### ⑧ 角色权限对比

**操作**：退出登录，用 `purchaser` 重新登录

**话术**：
> 切换为采购员账号后，左侧菜单只剩"首页仪表盘"和"采购订单"两项。
> 所有操作均记录到审计日志，可在管理员"审计日志"页面追溯。

**操作**：展示审计日志页面

## 🎯 老师可能问的问题与应答

**Q1：为什么从单文件 React 改为前后端分离？**

A：原版使用 IndexedDB 模拟数据库，无法体现真实数据库设计。新版采用 Spring Boot + MySQL 8，17 张表严格对应论文 E-R 设计，更贴近生产环境架构。

**Q2：TCO 公式的具体推导？**

A：参考论文 6.5.2 节：
- 维修方案 TCO = 修复费 + Σ(年保养费 × N) + Σ(年维修费 × N)
- 更换方案 TCO = 新设备价 + Σ(年保养费 × N) - 残值
- 残值估算 = 购置价 × (1 - 已用年限/设计寿命) × 0.70

**Q3：α 和 β 系数怎么定的？**

A：
- α 气候系数：寒冷 1.30、严寒 1.50、温和 1.00、湿热 1.10、沿海 1.20
- β AADT 系数：<5000→1.0、5000-10000→1.1、10000-20000→1.2、20000-30000→1.3、>30000→1.5
- 实际保养周期 = 标准周期 × α × β

**Q4：数据库表关系？**

A：
- bridge → device_archive（一座桥有多台设备）
- device_archive → lifecycle_event（一台设备有多个生命周期事件）
- device_archive → fault_order（一台设备有多个故障工单）
- device_archive → maintain_record（一台设备有多次保养记录）

**Q5：如何保证统计率的真实性？**

A：所有 8 项指标均由后端 StatsService 从数据库聚合计算：
```java
// 设备在用率
inUseRate = deviceMapper.selectCount(
    new LambdaQueryWrapper<DeviceArchive>()
        .eq(DeviceArchive::getStatus, "in_use")
) / totalCount * 100;
```

## 📌 关键术语对照

| 中文 | 英文 | 含义 |
|------|------|------|
| 全寿命周期成本 | TCO (Total Cost of Ownership) | 设备从购置到报废的全部成本 |
| 平均无故障时间 | MTBF (Mean Time Between Failures) | 故障间隔时长 |
| 平均修复时间 | MTTR (Mean Time To Repair) | 单次维修平均耗时 |
| 年平均日交通量 | AADT (Annual Average Daily Traffic) | 桥梁运行强度核心指标 |
| 角色权限控制 | RBAC (Role-Based Access Control) | 基于角色的访问控制 |
